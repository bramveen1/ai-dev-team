# M365 Email Provider — Delegate Access (Draft-Only)

## Overview

The M365 email provider gives an agent delegate access to a mailbox owner's Microsoft 365 inbox via the Microsoft Graph API. The agent can read messages and create/update/delete drafts, but **cannot send** on the owner's behalf. This enforces the trust boundary at three levels:

1. **Azure AD scopes**: Only `Mail.Read.Shared` and `Mail.ReadWrite.Shared` — no `Mail.Send`
2. **MCP tool surface**: The MCP server exposes no `send` tool — the LLM has no tool to call
3. **Permission framework**: The capability config has no `send` permission — the approval renderer shows "Open in Outlook" instead of a Send button

## Azure AD App Registration

### Step 1: Register the app

1. Go to [Azure Portal > App registrations](https://portal.azure.com/#blade/Microsoft_AAD_RegisteredApps/ApplicationsListBlade)
2. Click **New registration**
3. Name: `ai-dev-team-m365-delegate`
4. Supported account types: **Single tenant** (this organization only)
5. Redirect URI: Leave blank (we use device code flow)
6. Click **Register**

### Step 2: Configure API permissions

Add these **delegated** permissions under Microsoft Graph:

| Permission | Type | Purpose |
|---|---|---|
| `Mail.Read.Shared` | Delegated | Read messages in delegated mailboxes |
| `Mail.ReadWrite.Shared` | Delegated | Create, update, delete drafts in delegated mailboxes |
| `User.Read` | Delegated | Read user profile (required for Graph API auth) |
| `offline_access` | Delegated | Obtain refresh tokens for long-lived access |

**Do NOT add:**
- `Mail.Send` — explicitly excluded to enforce the no-send trust boundary
- `Mail.Send.Shared` — explicitly excluded
- Any application-level permissions — use delegated only

### Step 3: Create a client secret

1. Go to **Certificates & secrets** in the app registration
2. Click **New client secret**, add a description, set expiry
3. Copy the secret **value** (not the Secret ID) — store it in the secrets store

### Step 4: Enable public client flow

1. Go to **Authentication** in the app registration
2. Under **Advanced settings**, set **Allow public client flows** to **Yes**
3. Save

This enables the device code flow for headless authentication.

### Step 5: Grant admin consent (if required)

Depending on your tenant policy, a tenant admin may need to grant consent for the shared mailbox permissions.

### Step 6: Configure delegate access in Exchange

The agent's service account needs delegate access to the mailbox owner's inbox:

```powershell
# In Exchange Online PowerShell
Add-MailboxPermission -Identity owner@example.com -User agent-service@example.com -AccessRights ReadPermission
Add-MailboxFolderPermission -Identity owner@example.com:\Drafts -User agent-service@example.com -AccessRights Editor
```

## Authentication

### Prerequisites

Collect these values from the Azure AD app registration:
- **Tenant ID** — from the Overview page
- **Client ID** (Application ID) — from the Overview page
- **Client Secret** — from Certificates & secrets

Store them in `config/secrets/m365.json` (see [Secrets Store](../secrets-store.md)).

### Device code flow (initial setup)

The agent can initiate the device code flow interactively via Slack:

1. The agent asks the mailbox owner for the Azure AD app details (Tenant ID, Client ID, Client Secret)
2. The agent initiates the device code request:

```bash
curl -X POST "https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/devicecode" \
  -d "client_id={client_id}" \
  -d "client_secret={client_secret}" \
  -d "scope=Mail.Read.Shared Mail.ReadWrite.Shared User.Read offline_access"
```

3. The agent posts the user code and verification URL to Slack (e.g. "Visit https://microsoft.com/devicelogin and enter code **L9J8LQCM7**")
4. The mailbox owner completes authentication in their browser
5. The agent polls the token endpoint until authorization completes:

```bash
curl -X POST "https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/token" \
  -d "client_id={client_id}" \
  -d "client_secret={client_secret}" \
  -d "grant_type=urn:ietf:params:oauth:grant-type:device_code" \
  -d "device_code={device_code}"
```

6. The agent stores the resulting `access_token` and `refresh_token` in the secrets store

### Token refresh (ongoing)

Access tokens expire after 60–90 minutes. The secrets store auto-refreshes using the refresh token:

```bash
curl -X POST "https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/token" \
  -d "client_id={client_id}" \
  -d "client_secret={client_secret}" \
  -d "grant_type=refresh_token" \
  -d "refresh_token={refresh_token}" \
  -d "scope=Mail.Read.Shared Mail.ReadWrite.Shared User.Read offline_access"
```

Refresh tokens last ~90 days. If a refresh token expires, the device code flow must be repeated.

### Token storage

Tokens are stored in `config/secrets/m365.json`:

```json
{
  "client_id": "...",
  "tenant_id": "...",
  "client_secret": "...",
  "scopes": "Mail.Read.Shared Mail.ReadWrite.Shared User.Read offline_access",
  "access_token": "eyJ0eX...",
  "refresh_token": "0.AYYA...",
  "expires_at": 1713200000
}
```

## MCP Server

Location: `mcps/m365_mail/`

### Tools exposed

| Tool | Description |
|---|---|
| `list_messages` | List messages from a mail folder (inbox, drafts, etc.) |
| `read_message` | Read a single message by ID |
| `create_draft` | Create a new draft or reply draft |
| `update_draft` | Update an existing draft |
| `delete_draft` | Delete a draft |
| `get_draft_url` | Get Outlook web app deep link to a draft |

**No `send` tool** — if the LLM tries to send, there is no tool to call.

### Environment variables

| Variable | Description |
|---|---|
| `M365_ACCESS_TOKEN` | OAuth2 access token (resolved from secrets store) |
| `M365_ACCOUNT` | Mailbox owner UPN (e.g. `owner@example.com`) |
| `M365_SCOPES` | Computed scopes (set automatically by the capability framework) |

## Approval Flow

1. The agent reads the mailbox owner's inbox and decides to reply
2. The agent calls `create_draft` → draft is created in the owner's Drafts folder
3. The agent posts a Slack approval message with:
   - **Open in Outlook** (primary) — deep link to the draft in Outlook web
   - **Redraft** — ask the agent to revise the draft
   - **Discard** — delete the draft from M365 and mark as discarded
4. The mailbox owner clicks "Open in Outlook" → opens the draft in Outlook web/desktop
5. The mailbox owner reviews and sends manually from Outlook
6. The agent detects the send via message-status change in the next session (reconciliation)

## Capability Config

```yaml
<agent_name>:
  email:
    - instance: <owner_instance>
      provider: m365-mcp
      account: owner@example.com
      ownership: delegate
      permissions: [read, draft-create, draft-update, draft-delete]
```

Note: No `send` permission. The permission framework automatically renders the "Open in Outlook" flow instead of a Send button.

## Provider Config

```yaml
m365-mcp:
  command: python
  args: ["-m", "mcps.m365_mail"]
  capabilities: [email, calendar]
  permission_scopes:
    email:
      read: "Mail.Read.Shared"
      draft-create: "Mail.ReadWrite.Shared"
      draft-update: "Mail.ReadWrite.Shared"
      draft-delete: "Mail.ReadWrite.Shared"
      # No send scope
  secrets_map:
    M365_ACCESS_TOKEN: "m365:access_token"
  oauth:
    authority: "https://login.microsoftonline.com"
    token_path: "/oauth2/v2.0/token"
    devicecode_path: "/oauth2/v2.0/devicecode"
```
