# M365 Email Provider — Delegate Access (Draft-Only)

## Overview

The M365 email provider gives Lisa delegate access to Bram's Microsoft 365 inbox via the Microsoft Graph API. Lisa can read messages and create/update/delete drafts, but **cannot send** on Bram's behalf. This enforces the trust boundary at three levels:

1. **Azure AD scopes**: Only `Mail.Read.Shared` and `Mail.ReadWrite.Shared` — no `Mail.Send`
2. **MCP tool surface**: The MCP server exposes no `send` tool — the LLM has no tool to call
3. **Permission framework**: The capability config has no `send` permission — the approval renderer shows "Open in Outlook" instead of a Send button

## Azure AD App Registration

### Step 1: Register the app

1. Go to [Azure Portal > App registrations](https://portal.azure.com/#blade/Microsoft_AAD_RegisteredApps/ApplicationsListBlade)
2. Click **New registration**
3. Name: `ai-dev-team-m365-delegate`
4. Supported account types: **Single tenant** (this organization only)
5. Redirect URI: Leave blank (we use device code flow)
6. Click **Register**

### Step 2: Configure API permissions

Add these **delegated** permissions under Microsoft Graph:

| Permission | Type | Purpose |
|---|---|---|
| `Mail.Read.Shared` | Delegated | Read messages in delegated mailboxes |
| `Mail.ReadWrite.Shared` | Delegated | Create, update, delete drafts in delegated mailboxes |
| `User.Read` | Delegated | Read user profile (required for Graph API auth) |
| `offline_access` | Delegated | Obtain refresh tokens for long-lived access |

**Do NOT add:**
- `Mail.Send` — explicitly excluded to enforce the no-send trust boundary
- `Mail.Send.Shared` — explicitly excluded
- Any application-level permissions — use delegated only

### Step 3: Enable public client flow

1. Go to **Authentication** in the app registration
2. Under **Advanced settings**, set **Allow public client flows** to **Yes**
3. Save

This enables the device code flow for headless authentication.

### Step 4: Grant admin consent (if required)

Depending on your tenant policy, a tenant admin may need to grant consent for the shared mailbox permissions.

### Step 5: Configure delegate access in Exchange

Lisa's service account needs delegate access to Bram's mailbox:

```powershell
# In Exchange Online PowerShell
Add-MailboxPermission -Identity bram@pathtohired.com -User lisa-service@pathtohired.com -AccessRights ReadPermission
Add-MailboxFolderPermission -Identity bram@pathtohired.com:\Drafts -User lisa-service@pathtohired.com -AccessRights Editor
```

## Authentication

### Device code flow (initial setup)

```bash
# Request device code
curl -X POST "https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/devicecode" \
  -d "client_id={client_id}" \
  -d "scope=Mail.Read.Shared Mail.ReadWrite.Shared User.Read offline_access"

# User visits https://microsoft.com/devicelogin and enters the code
# Then exchange for tokens:
curl -X POST "https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/token" \
  -d "client_id={client_id}" \
  -d "grant_type=urn:ietf:params:oauth:grant-type:device_code" \
  -d "device_code={device_code}"
```

### Token refresh (ongoing)

```bash
curl -X POST "https://login.microsoftonline.com/{tenant_id}/oauth2/v2.0/token" \
  -d "client_id={client_id}" \
  -d "grant_type=refresh_token" \
  -d "refresh_token={refresh_token}" \
  -d "scope=Mail.Read.Shared Mail.ReadWrite.Shared User.Read offline_access"
```

Store tokens at `memory/systems/outlook-tokens.json`.

## MCP Server

Location: `mcps/m365_mail/`

### Tools exposed

| Tool | Description |
|---|---|
| `list_messages` | List messages from a mail folder (inbox, drafts, etc.) |
| `read_message` | Read a single message by ID |
| `create_draft` | Create a new draft or reply draft |
| `update_draft` | Update an existing draft |
| `delete_draft` | Delete a draft |
| `get_draft_url` | Get Outlook web app deep link to a draft |

**No `send` tool** — if the LLM tries to send, there is no tool to call.

### Environment variables

| Variable | Description |
|---|---|
| `M365_ACCESS_TOKEN` | OAuth2 access token |
| `M365_ACCOUNT` | Mailbox owner UPN (e.g. `bram@pathtohired.com`) |
| `M365_SCOPES` | Computed scopes (set automatically by the capability framework) |

## Approval Flow

1. Lisa reads Bram's inbox and decides to reply
2. Lisa calls `create_draft` → draft is created in Bram's Drafts folder
3. Lisa posts a Slack approval message with:
   - **Open in Outlook** (primary) — deep link to the draft in Outlook web
   - **Redraft** — ask Lisa to revise the draft
   - **Discard** — delete the draft from M365 and mark as discarded
4. Bram clicks "Open in Outlook" → opens the draft in Outlook web/desktop
5. Bram reviews and sends manually from Outlook
6. Lisa detects the send via message-status change in the next session (reconciliation)

## Capability Config

```yaml
lisa:
  email:
    - instance: bram
      provider: m365-mcp
      account: bram@pathtohired.com
      ownership: delegate
      permissions: [read, draft-create, draft-update, draft-delete]
```

Note: No `send` permission. The permission framework automatically renders the "Open in Outlook" flow instead of a Send button.

## Provider Config

```yaml
m365-mcp:
  command: python
  args: ["-m", "mcps.m365_mail"]
  capabilities: [email, calendar]
  permission_scopes:
    email:
      read: "Mail.Read.Shared"
      draft-create: "Mail.ReadWrite.Shared"
      draft-update: "Mail.ReadWrite.Shared"
      draft-delete: "Mail.ReadWrite.Shared"
      # No send scope
```
